# Walking Skeleton — LumaScope

**Phase:** 1
**Generated:** 2026-06-22

## Capability Proven End-to-End

> A developer can build LumaScope as Windows VST3 and standalone products, open the same embedded React/Material UI shell, and see a successful versioned handshake between JavaScript and the JUCE host.

## Architectural Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Native framework | JUCE 8.0.14, C++20, CMake 3.22+ | Current approved JUCE release with official VST3/Standalone and WebView2 support |
| Windows baseline | Windows 10 22H2 x64, VS2019 16.11 first verified toolchain | Matches the user's development machine; compatibility is proven before scaffold growth |
| Frontend | React + TypeScript + Vite + Material UI, pinned by npm lockfile | Locked modern WebView UI stack with deterministic local builds |
| Web assets | Vite `dist/` archived and embedded via `juce_add_binary_data`; served by JUCE `ResourceProvider` | Handles generated asset sets without configure-time file enumeration and works offline |
| Native/web contract | Protocol v1 handshake through JUCE native integration, event listeners, initialization data, and typed browser adapter | Gives later phases a stable, testable boundary |
| Development mode | Explicit localhost Vite URL via CMake cache/preset option; embedded assets remain default | Hot reload cannot silently alter release behavior |
| Codex documentation | Project `.codex/config.toml` contains one Context7 HTTP server; `AGENTS.md` records three library IDs | Portable project guidance without duplicate MCP servers |
| Test strategy | CTest plus Vitest/React Testing Library, orchestrated by PowerShell quick/full scripts | Fast task feedback and full native/web checks on Windows |
| Directory layout | Root build/scripts, `plugin/` native code, `ui/` web app, `tests/` fixtures | Keeps product boundaries explicit for later DSP, capture, and licensing phases |

## Stack Touched in Phase 1

- [ ] Project scaffold — CMake presets, pinned dependencies, npm workspace, lint/test runners.
- [ ] Native entry points — VST3 and standalone targets generated by `juce_add_plugin`.
- [ ] UI — one real React shell with bridge-driven connection state and retry/copy-diagnostics actions.
- [ ] Native/web read and write — browser emits `ui.ready@1`; native validates and emits/returns `host.info@1`; UI renders the returned product/mode/source state.
- [ ] Asset delivery — production UI is compiled, archived, embedded, and served through a path-safe resource provider.
- [ ] Development run — documented embedded and Vite commands exercise the complete local native/web path.
- [ ] Database — not applicable; Phase 1 and the desktop product shell have no persistence requirement. The required vertical state round-trip is the native/web protocol handshake.
