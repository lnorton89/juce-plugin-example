# 06.5-02-SUMMARY: Native, Worker, Infrastructure, and Script Config Extraction

**Completed:** 2026-06-24

## What Was Built

### Native Generated Config
- Added `cmake/templates/ProjectConfig.h.in`.
- Root CMake generates `generated/include/LumaScope/ProjectConfig.h` from `project-config.json` values.
- `plugin/CMakeLists.txt` now uses config-derived plugin target, company name, bundle ID, manufacturer code, plugin code, and product name.
- Native sources now consume `config::productName`, `config::companyName`, `config::appdataDirName`, `config::sourcePreferenceFilename`, `config::entitlementFilename`, `config::devServerUrl`, and configured fallback copy.

### Build/Test Config Propagation
- Root `CMakeLists.txt` configures the native header once at build-root scope.
- `tests/native/CMakeLists.txt` uses `${PROJECT_CONFIG_TEST_TARGET}`, `${PROJECT_CONFIG_NAME}`, and `${PROJECT_CONFIG_VERSION_STR}`.
- `cmake/WebBundle.cmake` derives bundle target names from config while preserving the generated C++ header name used by existing includes.

### Worker Config
- Added `worker/src/config/projectConfig.ts`.
- `ActivationSuccessResponse.tokenType` now uses `EntitlementTokenType` from config.
- Activation route responses use `projectConfig.tokenTypeV1` instead of a local literal.
- Worker typecheck and tests pass with the config import path.

### PowerShell Config
- Added `cmake/templates/config.ps1.in` and generated-style `scripts/config.ps1`.
- `scripts/generate-config-ts.mjs` regenerates PowerShell config alongside UI and Worker modules.
- Bootstrap, quick test, full test, and standalone launcher scripts dot-source `scripts/config.ps1` for target names and runtime paths.

### Infrastructure Traceability
- `worker/wrangler.toml` and `infra/manifest.yaml` now explicitly identify `project-config.json` as the source of truth for logical resource names.

## Deviations

- The generated native header keeps `PROJECT_NAMESPACE` support available, but the full namespace rename sweep is intentionally not performed because it would require moving include paths and class names far beyond the safe config-centralization slice.
- `wrangler.toml` and `infra/manifest.yaml` remain checked-in example configuration files, with comments tying logical names back to `project-config.json`. They are not generated on every run because Wrangler expects concrete TOML in version control.
