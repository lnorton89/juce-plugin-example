# Phase 6.5: Project Configuration Centralization — Research

**Researched:** 2026-06-23
**Domain:** Cross-language configuration extraction, CMake build tooling, code generation, JSON Schema
**Confidence:** HIGH

## Summary

This phase extracts ~300+ hardcoded values across C++, TypeScript, CMake, YAML, TOML, PowerShell, and Markdown into a single `project-config.json` at the project root, validated by `project-config.schema.json`. CMake 4.0.3 (installed) has full native JSON parsing via `string(JSON ...)` since 3.19, so no external scripts are needed for CMake-side consumption. C++ headers are generated via `configure_file()` with `@ONLY` substitution. TypeScript imports JSON directly via `resolveJsonModule` or through a code-generated `.ts` module. PowerShell scripts consume a generated `config.ps1` module via dot-sourcing.

The most complex challenge is the C++ `lumascope` namespace and `LumaScope/` include directory — these appear 31+ and 60+ times respectively across code and headers. A pragmatic approach: generate a `ProjectConfig.h` with `constexpr` values, use macro aliases for the namespace, and accept that directory-level rename (`LumaScope/` → `${PROJECT_NAME}/`) is done by renaming the directory once and updating include paths in CI.

**Primary recommendation:** Use CMake's native `string(JSON ...)` to read `project-config.json` at configure time. Generate C++ `ProjectConfig.h` and a `config.ps1` via `configure_file()`. TypeScript and Worker code import JSON or a generated `.ts` module. Validate early with `message(FATAL_ERROR)` for missing/invalid config values.

**Key insight:** This isn't just a find-and-replace task — the JSON Schema with `filesAffected` per value is the real value for future maintainers. The code generation is mechanical; the schema design is where the architectural thinking lives.

### Pattern 1: CMake JSON Parsing

**What:** Use CMake's native `string(JSON ...)` command (available since CMake 3.19, verified with 4.0.3) to read typed values from `project-config.json` at configure time.

**When to use:** Every CMakeLists.txt that currently references hardcoded product names, versions, codes, or paths.

**Example:**
```cmake
# cmake/Config.cmake — reads project-config.json and sets CMake variables
file(READ "${CMAKE_SOURCE_DIR}/project-config.json" _CONFIG_JSON)

# Helper function for validated reads
function(read_config VALUE PATH_SPEC)
  string(JSON "${VALUE}" ERROR_VARIABLE _ERR GET "${_CONFIG_JSON}" ${PATH_SPEC})
  if("${_ERR}" MATCHES ".*not found")
    message(FATAL_ERROR "project-config.json is missing '${PATH_SPEC}'. "
                        "See project-config.schema.json for required values.")
  endif()
  if("${${VALUE}}" STREQUAL "")
    message(FATAL_ERROR "project-config.json '${PATH_SPEC}' must not be empty. "
                        "Set it in ${CMAKE_SOURCE_DIR}/project-config.json")
  endif()
  set("${VALUE}" "${${VALUE}}" PARENT_SCOPE)
endfunction()

# Read top-level values
read_config(PROJECT_CONFIG_NAME "product" "name")                 # e.g., "LumaScope"
read_config(PROJECT_CONFIG_COMPANY "product" "companyName")      # e.g., "Signal Foundry Audio"
read_config(PROJECT_CONFIG_VERSION_STR "version" "string")       # e.g., "0.1.0"
read_config(PROJECT_CONFIG_NAMESPACE "identifiers" "namespace")  # e.g., "lumascope"

# Read nested values from different branches
read_config(PROJECT_CONFIG_BUNDLE_ID "identifiers" "bundleId")   # e.g., "audio.signal-foundry.LumaScope"
read_config(PROJECT_CONFIG_MFR_CODE "identifiers" "plugin" "manufacturerCode")  # e.g., "SgFd"
read_config(PROJECT_CONFIG_PLUGIN_CODE "identifiers" "plugin" "pluginCode")       # e.g., "LmSc"
```

Source: CMake 4.0.3 documentation — `string(JSON ...)` available since 3.19 [VERIFIED: cmake --version and cmake --help-full]
[ASSUMED: detailed error handling pattern — CMake string(JSON) error variable behavior may differ in edge cases]

### Pattern 2: configure_file() for C++ Header Generation

**What:** Use CMake's `configure_file()` with `@ONLY` to substitute `@VARIABLE@` placeholders in a `.h.in` template file. The output is a header included by C++ sources.

**When to use:** For generating `ProjectConfig.h` that all C++ sources include to access product identity constants.

**Example template (`cmake/templates/ProjectConfig.h.in`):**
```cpp
#pragma once

// Generated from project-config.json — do not edit manually.
// Edit project-config.json and reconfigure CMake to regenerate.

// Nested namespace using preprocessor stringification
#define PROJECT_NAMESPACE @PROJECT_CONFIG_NAMESPACE@
#define PROJECT_NAMESPACE_STR "@PROJECT_CONFIG_NAMESPACE@"

namespace @PROJECT_CONFIG_NAMESPACE@ {
namespace config {

// Product identity
inline constexpr auto productName       = "@PROJECT_CONFIG_NAME@";
inline constexpr auto companyName       = "@PROJECT_CONFIG_COMPANY@";
inline constexpr auto version           = "@PROJECT_CONFIG_VERSION_STR@";

// Identifiers
inline constexpr auto bundleId          = "@PROJECT_CONFIG_BUNDLE_ID@";
inline constexpr auto appdataDirName    = "@PROJECT_CONFIG_APPDATA_DIR@";
inline constexpr auto copyright         = "Copyright (c) @PROJECT_CONFIG_YEAR@ @PROJECT_CONFIG_COMPANY@";

// URLs
inline constexpr auto devServerUrl      = "@PROJECT_CONFIG_DEV_SERVER_URL@";
inline constexpr auto apiUrl            = "@PROJECT_CONFIG_API_URL@";

// Cloud
inline constexpr auto tokenTypeV1       = "@PROJECT_CONFIG_TOKEN_TYPE_V1@";

// Build
inline constexpr auto pluginCode        = "@PROJECT_CONFIG_PLUGIN_CODE@";
inline constexpr auto manufacturerCode  = "@PROJECT_CONFIG_MFR_CODE@";

} // namespace config
} // namespace @PROJECT_CONFIG_NAMESPACE@
```

**CMake invocation:**
```cmake
configure_file(
  "${CMAKE_SOURCE_DIR}/cmake/templates/ProjectConfig.h.in"
  "${CMAKE_CURRENT_BINARY_DIR}/generated/ProjectConfig.h"
  @ONLY
)
target_include_directories(LumaScope PRIVATE "${CMAKE_CURRENT_BINARY_DIR}/generated")
```

**C++ usage:**
```cpp
#include "ProjectConfig.h"
// Use: lumascope::config::productName or PROJECT_NAMESPACE
```

Source: CMake `configure_file()` documentation [VERIFIED: CMake 3.22+ docs]

### Pattern 3: C++ Namespace Aliasing via Macro

**What:** Use the `#define PROJECT_NAMESPACE lumascope` pattern from the generated header to allow all source files to reference the namespace by macro rather than hardcoded name. This avoids touching every `namespace lumascope { ... }` block in source files — keep them as `namespace PROJECT_NAMESPACE { ... }` via macro expansion.

**When to use:** For namespace declarations and all `lumascope::` qualified references.

**Example:**
```cpp
// In generated ProjectConfig.h:
#define PROJECT_NAMESPACE lumascope
#define PROJECT_NAMESPACE_STR "lumascope"

// In source files (replace `namespace lumascope`):
namespace PROJECT_NAMESPACE
{
    // ... existing code, unchanged except the keyword ...
}

// For out-of-namespace references (replace `lumascope::X`):
PROJECT_NAMESPACE::HostBridge bridge;
```

**Risks:**
- C++ identifiers can't be used with `#define` in all positions — `namespace PROJECT_NAMESPACE` works because macros expand before parsing, but preprocessor stringification and concatenation have `##` token-pasting caveats.
- `PROJECT_NAMESPACE::` in header files works because the macro is defined before inclusion.
- The `LumaScope` class prefix (e.g., `LumaScopeAudioProcessor`) is harder to macro-ize because it's used in header guards, constructors, destructors, etc. A separate `PROJECT_CLASS_PREFIX` macro can handle this.

Source: Standard C++ preprocessor technique [CITED: cppreference.com/preprocessor/replace]
[ASSUMED: practical experience — no official documentation covers this pattern for JUCE projects]

### Pattern 4: TypeScript Config Generation

**What:** Generate `ui/src/config/projectConfig.ts` from `project-config.json` using a small Node.js script. This enables tree-shaking (only imported values end up in the bundle) and provides TypeScript types.

**When to use:** For all UI code that needs product name, company name, version, or branding.

**Example generation script (`scripts/generate-config-ts.mjs`):**
```javascript
import { readFileSync, writeFileSync } from 'fs';
const config = JSON.parse(readFileSync('project-config.json', 'utf-8'));

const output = `// Generated from project-config.json — do not edit manually.
// Run: node scripts/generate-config-ts.mjs

export const productName = ${JSON.stringify(config.product.name)};
export const companyName = ${JSON.stringify(config.product.companyName)};
export const version = ${JSON.stringify(config.version.string)};
export const appdataDirName = ${JSON.stringify(config.paths.appdataDirName)};

export const urls = {
  devServer: ${JSON.stringify(config.urls.devServer)},
  api: ${JSON.stringify(config.urls.productionApi)},
} as const;

export const errorTemplates = {
  webview2Missing: ${JSON.stringify(`The WebView2 runtime is unavailable. Install or repair Evergreen WebView2, then reopen ${config.product.name}.`)},
  resourceMissing: ${JSON.stringify(`The packaged interface could not be loaded. Rebuild ${config.product.name} and reopen it.`)},
} as const;
`;

writeFileSync('ui/src/config/projectConfig.ts', output);
```

**Alternative — direct JSON import (simpler):**
```typescript
// tsconfig.json must have: "resolveJsonModule": true
import config from '../../../project-config.json' assert { type: 'json' };
const { name: productName } = config.product;
```

Source: TypeScript `resolveJsonModule` is a standard tsconfig option [VERIFIED: www.typescriptlang.org/tsconfig#resolveJsonModule]

### Pattern 6: PowerShell Config Module Generation

**What:** Generate `scripts/config.ps1` via CMake `configure_file()` with `@ONLY` substitution. Scripts dot-source this file to get product name, paths, and build targets.

**When to use:** For all `.ps1` scripts that reference hardcoded target names, executable paths, or product names.

**Example template (`cmake/templates/config.ps1.in`):**
```powershell
# Generated from project-config.json — do not edit manually.
$script:PRODUCT_NAME = "@PROJECT_CONFIG_NAME@"
$script:COMPANY_NAME = "@PROJECT_CONFIG_COMPANY@"
$script:VERSION = "@PROJECT_CONFIG_VERSION_STR@"
$script:STANDALONE_TARGET = "@PROJECT_CONFIG_STANDALONE_TARGET@"
$script:VST3_TARGET = "@PROJECT_CONFIG_VST3_TARGET@"
$script:TEST_TARGET = "@PROJECT_CONFIG_TEST_TARGET@"
$script:BUILD_DIR = "@PROJECT_CONFIG_DEFAULT_BUILD_DIR@"
$script:APPDATA_DIR_NAME = "@PROJECT_CONFIG_APPDATA_DIR@"

function Get-PluginTarget { return "${script:PRODUCT_NAME}_VST3" }
function Get-StandaloneTarget { return "${script:PRODUCT_NAME}_Standalone" }
function Get-NativeTestTarget { return "${script:PRODUCT_NAME}NativeTests" }
```

**PowerShell usage:**
```powershell
. "$PSScriptRoot/config.ps1"
$exePath = Join-Path $resolvedBuild "plugin\$Config\$($script:PRODUCT_NAME)_Standalone.exe"
```

Source: Standard PowerShell dot-sourcing pattern [CITED: docs.microsoft.com/en-us/powershell/module/microsoft.powershell.core/about/about_scripts]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON parsing in CMake | Python/Node helper script | CMake `string(JSON ...)` | CMake 4.0.3 has it built-in; adding external runtime dependency is unnecessary complexity |
| JSON Schema validation | Custom CMake validation logic | `ajv` CLI via `npx` | Schema is reusable for CI, editor autocompletion, and documentation generation |
| C++ string constants | Hand-written header with inline strings | `configure_file()` template | Automatic regeneration from config, one source of truth |
| PowerShell template substitution | Manual find-and-replace | CMake `configure_file()` with `@ONLY` | Same pattern as C++ header generation, zero new tooling |
| TypeScript config ingestion | Manual sync of constants | Generated `.ts` module or direct JSON import | Compile-time type checking, tree-shakeable |

**Key insight:** The temptation is to build custom tooling for every target language. The smarter approach is to use each ecosystem's native config-reading mechanism (CMake `string(JSON)`, TypeScript `resolveJsonModule`, Worker JSON import, PowerShell dot-sourcing) with one generation pass for C++ (which has no runtime JSON reading capability at compile time).

## Common Pitfalls

### Pitfall 1: Namespace Macro Collisions
**What goes wrong:** Using `#define NAMESPACE lumascope` works for `namespace NAMESPACE { }` blocks but breaks when `NAMESPACE::` is used in contexts where the preprocessor doesn't expand (e.g., in string concatenation or template arguments).
**Why it happens:** The preprocessor only does token-level replacement, not contextual expansion.
**How to avoid:** Define both a macro for namespace declaration and a separate string constant. Test with one source file first. Avoid `NAMESPACE::X` in macros that are themselves concatenated with `##`.
**Warning signs:** Compiler errors about "not a class or namespace" when using namespace alias.

### Pitfall 2: configure_file() Escaping Issues
**What goes wrong:** `configure_file()` treats `@VAR@` as the CMake variable `VAR`. If the value contains characters special to CMake or the target language (e.g., `#`, `\`, quotes), the output may be malformed.
**Why it happens:** CMake does variable substitution without escaping for the target language.
**How to avoid:** Use `@ONLY` to prevent `${}` substitution. For values containing quotes (e.g., `#include` guards), use CMake's `string(MAKE_C_IDENTIFIER ...)` or wrap in separate replacement passes. Test generated files after first build.
**Warning signs:** Generated header produces compiler errors on string literals that appear truncated or contain extra backslashes.

### Pitfall 3: Incomplete filesAffected Documentation
**What goes wrong:** The `filesAffected` array in the schema lists only the obvious files, missing edge cases like test fixtures, documentation examples, or stack trace references.
**Why it happens:** It's time-consuming to hunt every reference, and some are dynamic (e.g., log messages constructed at runtime).
**How to avoid:** After initial schema creation, run a comprehensive grep for each value across ALL file types (not just source code). Add a CI step that flags config values appearing in untracked file paths.
**Warning signs:** A rebranding exercise (search-and-replace all config values) reveals files not listed in `filesAffected`.

### Pitfall 4: Build Target Name Changes Break CI Scripts
**What goes wrong:** Changing CMake target names (e.g., `LumaScope_Standalone` → `${PRODUCT_NAME}_Standalone`) causes CI scripts to fail because they reference old names.
**Why it happens:** PowerShell and YAML scripts hardcode target names that are derived from the CMake target, not the config.
**How to avoid:** Make ALL build target names config-driven. Both `juce_add_plugin(targetName ...)` and `add_dependencies(...)` must use config-derived names. Generate or export target names to a `config.ps1` that CI scripts consume.
**Warning signs:** `cmake --build --target` commands fail after config change, CI cannot find built artifacts.

### Pitfall 5: Binary Data Namespace Mismatch
**What goes wrong:** `juce_add_binary_data(LumaScopeWebBundle NAMESPACE LumaScopeWebBundleData ...)` generates C++ code using the namespace `LumaScopeWebBundleData::lumascopeweb_zip`. Changing the namespace causes compilation errors in files that include the generated header.
**Why it happens:** `juce_add_binary_data` generates C++ based on its arguments; the consumer (`WebResources.cpp`) references the generated symbols.
**How to avoid:** Make the binary data target name and namespace config-driven. `WebResources.cpp` should `#include "ProjectConfig.h"` and use config-derived namespace references.
**Warning signs:** Linker errors about `LumaScopeWebBundleData::lumascopeweb_zip` not found.

### Validation Script (validate-config.mjs)

```javascript
#!/usr/bin/env node
import { readFileSync } from 'fs';
import Ajv from 'ajv';

const ajv = new Ajv();
const schema = JSON.parse(readFileSync('project-config.schema.json', 'utf-8'));
const config = JSON.parse(readFileSync('project-config.json', 'utf-8'));

const validate = ajv.compile(schema);
const valid = validate(config);
if (!valid) {
  console.error('❌ project-config.json validation failed:');
  for (const err of validate.errors) {
    console.error(`  ${err.instancePath || '(root)'}: ${err.message}`);
  }
  process.exit(1);
}

// Semantic checks
const versionMatch = /^(\d+)\.(\d+)\.(\d+)$/.exec(config.version.string);
if (!versionMatch) {
  console.error('❌ version.string must be in major.minor.patch format');
  process.exit(1);
}

if (config.identifiers.plugin.manufacturerCode.length !== 4) {
  console.error('❌ identifiers.plugin.manufacturerCode must be exactly 4 characters');
  process.exit(1);
}

console.log('✅ project-config.json is valid');
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Hardcoded strings in source files | Single JSON config with schema | Phase 6.5 | One JSON file to edit, schemas to validate, auto-generated constants |
| Projucer-based project config in JUCE | CMake-native config via `juce_add_plugin()` | JUCE 6.0+ | CMake reads all JUCE target properties; no Projucer needed |
| CMake `file(READ)` + regex for JSON | CMake `string(JSON ...)` | CMake 3.19 (2019) | Safe structured parsing, no regex fragility |
| Per-file constants in TypeScript | `resolveJsonModule` + direct import | TypeScript 2.9 (2018) | Zero-code config ingestion, type-safe |

**Deprecated/outdated:**
- **CMake `file(READ)` + manual string parsing**: Replaced by `string(JSON ...)` which handles nesting, escaping, and errors natively since CMake 3.19.
- **Python/Node JSON bridge scripts for CMake**: Unnecessary when CMake 4.0.3 is installed with full JSON support.

## Open Questions (RESOLVED)

1. **Include directory rename strategy**
   - What we know: `plugin/include/LumaScope/` directory is referenced in 44+ `#include` paths in source files and 16+ in headers. Many include paths are also in `target_sources()` in CMake.
   - What's unclear: Should the directory be renamed to `${config.name}` or stay as-is and documented as a manual rename target? Renaming breaks every include path.
    - Recommendation: Keep the directory name as `LumaScope/` for now (it's a project-internal implementation detail). Document in the schema's `filesAffected` that renaming requires updating 60+ include paths. Defer automated directory rename to v2. (RESOLVED: keep + document, defer rename)

2. **Binary data namespace in CMake**
   - What we know: `juce_add_binary_data(LumaScopeWebBundle NAMESPACE LumaScopeWebBundleData ...)` generates `LumaScopeWebBundleData::lumascopeweb_zip`. The `NAMESPACE` argument affects generated C++ identifiers.
   - What's unclear: Whether `NAMESPACE` can be a CMake variable or must be a literal string. JUCE's CMake API documentation doesn't say explicitly.
    - Recommendation: Test with `${PROJECT_CONFIG_WEB_BUNDLE_NAMESPACE}` in plan phase. If it works, great. If not, wrap in a generator expression. (RESOLVED: use CMake variable, test in plan phase)

3. **Configure-time vs. Generate-time for .ps1**
   - What we know: CMake `configure_file()` runs at CMake configure time. This means `scripts/config.ps1` is generated in the source tree during CMake configuration.
   - What's unclear: Is it acceptable to have CMake write to the source tree (polluting `git status`)? Should `.ps1` files be generated in the build tree and loaded with an absolute path?
    - Recommendation: Generate in the build tree and have scripts accept a `-ConfigPath` parameter or discover it relative to the script directory via a symlink/known path. This keeps the source tree clean. (RESOLVED: build-tree generation, scripts discover via relative path)

4. **TypeScript literal type narrowing for productName**
   - What we know: Currently `productName: 'LumaScope'` is a literal type that acts as a discriminant in `HostInfo`. If productName becomes a variable, the literal type narrowing disappears.
   - What's unclear: Whether the `HostInfo.productName` type needs to remain a literal union for discriminated union narrowing, or if `string` is sufficient.
    - Recommendation: Keep a TypeScript `as const` type export that re-exports the config value as a literal type. This preserves type narrowing while being config-driven. (RESOLVED: as const re-export pattern)

5. **CMake target names and the `project()` command**
   - What we know: `project(LumaScope VERSION 0.1.0 LANGUAGES CXX)` sets the CMake project name. This is used by `PROJECT_VERSION` and `PROJECT_SOURCE_DIR` variables.
   - What's unclear: Can the `project()` name be any string, or does it affect generated JUCE code? If we make it config-driven, does `PROJECT_VERSION` still work?
    - Recommendation: Set `project()` name statically (it's an internal CMake identifier, not user-facing). Read version from config separately. Use `${PROJECT_CONFIG_VERSION_STR}` where user-facing versions are needed. (RESOLVED: static project() name, config-driven version separately)
