# Phase 6.5: Project Configuration Centralization — Context

**Gathered:** 2026-06-23
**Status:** Ready for planning
**Source:** User discussion + codebase exploration

<domain>
## Phase Boundary

Extract all hardcoded values currently scattered across the codebase into centralized, typed configuration files. Create schemas that document every configuration value and list every file location affected by it. Add build-time and validation tooling so the project can be rebranded, forked, or distributed by editing configuration files rather than hunting through source code.

This phase touches every layer of the project: C++ (JUCE plugin), TypeScript/React (UI), CMake (build), Worker (Cloudflare), infrastructure scripts (PowerShell), and documentation.

**What stays in scope:** All hardcoded values that change per-product-instance:
- Product name (LumaScope) and company name (Signal Foundry Audio)
- Version strings (0.1.0)
- JUCE plugin codes (SgFd, LmSc)
- Bundle IDs (audio.signal-foundry.LumaScope)
- C++ namespace (lumascope)
- API URLs and route patterns
- Worker/D1 resource names and env var contracts
- AppData directory paths
- User-facing error strings containing product name
- UI brand header text
- Bridge protocol identifiers (productName, companyName)
- Token type identifiers (lumascope-entitlement-v1)
- Build target names (LumaScope_VST3, LumaScope_Standalone, etc.)
- Script and documentation references

**What stays out of scope:**
- Pinned dependency *versions* (JUCE 8.0.14, MUI 9.1.1, etc.) — these are product-team decisions, not per-instance configuration
- Internal API version numbers (protocol v1, API v1) — protocol compatibility
- FFT and DSP constants (block sizes, window types, etc.) — algorithmic parameters
- Lemon Squeezy test fixture values (store_1, prod_1, var_1) — test data
- Rate limiting and timeout constants — operational tuning
</domain>

<decisions>
## Implementation Decisions

### Configuration Format
- **Main config**: `project-config.json` at the project root — versioned JSON with all configurable values
- **Schema**: `project-config.schema.json` — JSON Schema that validates the config AND documents every affected file path per value
- **Secondary configs**: Where logical grouping helps (e.g., `infra-config.json` for worker-specific values), but the main config is the single source of truth

### Config Value Categories
- **product**: name, company name, copyright, author
- **version**: current version, version string format
- **identifiers**: bundle ID, plugin codes (manufacturer, plugin), C++ namespace
- **paths**: appdata directory name, source preference filename, entitlement filename
- **urls**: dev server URL, production API URL, preview API URL
- **ui**: window title, brand header text, error message templates
- **bridge**: protocol product name, company name identifiers
- **worker**: worker names, D1 database names, route patterns, env var names
- **cloud**: Cloudflare resource names, route patterns, required secrets list
- **tokens**: token type identifiers for entitlements
- **build**: CMake target names, test target names, artifact names

### CI/Config Validation Approach
- Build-time CMake step reads `project-config.json` and fails early if required values are missing
- A Node.js script can regenerate all derived constants/headers for local development
- Config changes abort the build if the schema version doesn't match what consuming code expects

### Code Generation Strategy
- C++: CMake `configure_file()` or a small script generates `Config.h` from JSON values
- TypeScript: Vite import or a small script generates `config.ts` from JSON values
- CMake: Read JSON directly via CMake's `file(READ)` + string manipulation or `json_get()` in modern CMake
- Worker: Import JSON directly (ES modules can import JSON)
- Docs/scripts: Template variables replaced by build script

### Namespace Rename Consideration
- The C++ `lumascope` namespace should be driven by config
- All `#include` paths with `LumaScope/` directory need to derive from config
- This is the hardest extraction target — requires coordinated change across 13+ header files and all source files

### JSON Schema as Documentation
- Each config value's schema entry includes a `filesAffected` array listing every file that references it
- A separate `project-config-matrix.md` can be auto-generated from the schema showing the complete dependency graph

### Edge Cases
- User-facing error strings containing "LumaScope" should use format strings like `{productName}` at the bridge protocol level, filled in by the UI
- Documentation `.md` files should use template variables (e.g., `{{product_name}}`) substituted by a post-processing script or documented as manual replace targets
- AppData directory paths and entitlement filenames should derive from config
- Fallback canvas text and error diagnostics must use config-driven product name

### Configuration for Distribution
- The schema and config file are designed so a user can fork the repo, edit ~20-30 values in `project-config.json`, and rebuild a fully rebranded product
- A config validation script checks that all required values are non-empty before the build proceeds
- Missing or invalid config produces a clear error message telling the user what to set

### the agent's Discretion
- Exact JSON structure (flat vs. nested, key naming conventions)
- Whether to generate TypeScript config from the JSON or import it directly
- Whether to use CMake's built-in JSON parsing or a helper script
- Specific names of derived constants/types
- Whether to extract inline constants from C++ headers that are already clean (like `snapshotPollingMs`)
- Binary/includes directory naming for the `LumaScope/` include path — whether to rename directories or keep as config-driven

</decisions>

<specifics>
## Specific Ideas

1. **Config file location**: `project-config.json` at repo root, with JSON Schema at `project-config.schema.json`
2. **Validation**: CMake `configure` step that reads the JSON, validates against schema, and fails with clear error messages for missing values
3. **Generated C++ header**: CMake generates `plugin/include/LumaScope/ProjectConfig.h` from config — contains constexpr strings for product name, company name, version, URLs, appdata paths
4. **Generated TypeScript module**: Script generates `ui/src/config/projectConfig.ts` from config — exports product name, company name, version for UI consumption
5. **CMake variable injection**: CMake reads config and sets variables like `PROJECT_CONFIG_NAME`, `PROJECT_CONFIG_VERSION`, `PROJECT_CONFIG_BUNDLE_ID` used in target properties
6. **Worker config**: Worker imports JSON directly or through a generated TypeScript module with resource names
7. **Script templates**: Replace hardcoded names in .ps1 scripts with environment variables or a generated config.ps1 module
8. **Documentation templates**: Use `{{variable}}` syntax in `.md` files that get resolved by a build step, or add a pre-commit check
</specifics>
