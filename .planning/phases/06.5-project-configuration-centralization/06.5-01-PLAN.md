<objective>
**Fix:** Hardcoded product identity values are scattered across CMake files with no configuration schema or validation.

**What this plan delivers:** The foundational configuration file and schema, CMake integration to read config at configure time, and build-time validation tooling. After this plan, a developer can edit `project-config.json` and the CMake layer will read those values — and any invalid/missing config will produce clear build failures.

**MVP vertical slice:** After this plan, a developer can change the product name in one file, reconfigure CMake, and see `cmake -LA` report the new values via the Config.cmake variables. The config file is validated against its schema at configure time.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06.5-01 | Tampering | project-config.json | mitigate | Config file is versioned in git; tampering shows in `git diff`. Schema validation via `validate-config.mjs` at build time blocks malformed configs |
| T-06.5-02 | Elevation of Privilege | cmake/Config.cmake | mitigate | CMake `string(JSON ...)` parses structured JSON only — no arbitrary code execution. `message(FATAL_ERROR)` stops build on unexpected structure |
| T-06.5-03 | Information Disclosure | Generated headers/binaries | accept | Embedded config strings in compiled binaries are identical to previously hardcoded strings — no new disclosure risk |
| T-06.5-SC | Tampering | ajv/ajv-cli (npx) | mitigate | `npx --yes ajv-cli` uses npm registry. Well-known package with 50M+ weekly downloads. Fallback to basic validation if ajv unavailable |
</threat_model>
