# 06.5-03-SUMMARY: UI Config Generation, Documentation, and Regeneration Command

**Completed:** 2026-06-24

## What Was Built

### TypeScript Config Generation
- Added `scripts/generate-config-ts.mjs`.
- Added `scripts/generate-all-config.mjs` as the single regeneration command.
- Regeneration writes:
  - `ui/src/config/projectConfig.ts`
  - `worker/src/config/projectConfig.ts`
  - `scripts/config.ps1`
- `project-config.json` includes a `_comment` regeneration hint.

### UI Config Consumption
- `ui/src/bridge/protocol.ts` imports config-derived product and company literals while preserving TypeScript literal narrowing via `typeof projectConfig.productName` and `typeof projectConfig.companyName`.
- `BrandHeader`, `StatusFooter`, `BridgeStatus`, `ActivationDialog`, `DeactivationDialog`, and `GraceWarnAlert` use config-driven product/company copy.
- The UI config generator avoids embedding remote URLs in UI source, preserving the offline bundle contract.

### Package Scripts
- `ui/package.json` includes:
  - `generate-config`
  - `generate-all-config`
  - `validate-config`

### Documentation
- Added `docs/CONFIG-REFERENCE.md` documenting config categories, generated consumers, regeneration, and template variables.
- README now includes `{{product_name}}` and `{{version}}` template-variable conventions.

## Deviations

- Documentation is not fully templated end to end. The project now establishes the template-variable convention and reference doc; a later docs-specific pass can expand every narrative occurrence without mixing broad docs rewrite into the source extraction.
- UI config intentionally omits the WebView2 download URL from generated UI source so the offline/no-remote-resource bundle test remains valid.
